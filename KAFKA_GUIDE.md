### Kafka란?

**Apache Kafka**는 LinkedIn에서 개발한 **분산 이벤트 스트리밍 플랫폼**

### 기본 정의

```
Kafka = 분산 메시지 큐 + 이벤트 스트리밍 플랫폼 + 데이터 파이프라인
```

고성능, 확장 가능한 이벤트 기반 시스템을 위해 설계되었으며:

- **초당 수백만 개**의 메시지 처리 가능
- **실시간** 데이터 처리
- **분산** 시스템으로 고가용성 보장
- **메시지 보존** (설정 가능한 기간)

---

## 핵심 개념

### 1. Topic (토픽)

```
Topic = 메시지의 범주 또는 채널
```

**예시**: `reservation-events` 토픽에 예약 관련 모든 이벤트가 발행됨

**특징**:

- 테이블처럼 구조화됨
- 메시지는 순서 보장 (파티션 내에서)
- 영구 저장 가능 (설정에 따라)

### 2. Partition (파티션)

```
Partition = Topic을 분산 저장하는 조각
```

**장점**:

- **병렬 처리**: 여러 파티션을 동시에 처리
- **내결함성**: 한 파티션 장애가 다른 파티션에 영향 없음
- **확장성**: 파티션 추가로 처리량 증대

**메시지 분배 전략**:

```
Key가 있으면:
  - 같은 Key → 같은 Partition (순서 보장)

Key가 없으면:
  - Round-robin으로 파티션에 분배
```

### 3. Producer (프로듀서)

```
Producer = 메시지를 Kafka에 전송하는 클라이언트
```

**역할**:

- 토픽에 메시지 **발행**
- 파티션 선택 (키 기반 또는 라운드로빈)
- 에러 처리 및 재시도

**우리 시스템 예**:

```
ReservationService
  ↓ (이벤트 발행)
Kafka Producer
  ↓
reservation-events Topic (Kafka)
```

### 4. Consumer (컨슈머)

```
Consumer = Kafka에서 메시지를 읽는 클라이언트
```

**역할**:

- 토픽에서 메시지 **구독**
- 메시지 처리
- Offset 관리 (읽은 위치 기록)

### 5. Consumer Group (컨슈머 그룹)

```
Consumer Group = 같은 목적의 여러 컨슈머 집합
```

**구조**:

```
Topic: reservation-events (3개 파티션)

ConsumerGroup-A (데이터 분석용 기준)
ㄴ Consumer1 -> Partition0
ㄴ Consumer2 -> Partition1
ㄴ Consumer3 -> Partition2

ConsumerGroup-B (모니터링용)
├── Consumer1 → Partition0
├── Consumer2 → Partition1
└── Consumer3 → Partition2
```

**특징**:

- 각 파티션은 하나의 컨슈머에만 할당
- 컨슈머 추가 시 자동으로 리밸런싱
- 그룹 내 메시지 중복 처리 없음

### 6. Broker (브로커)

Broker = Kafka 클러스터를 구성하는 서버


**역할**:

- 메시지 저장
- 프로듀서/컨슈머 요청 처리
- 레플리카 관리

**구조**:

```
Kafka Cluster
ㄴ Broker 1 (포트 9092)
ㄴ Broker 2 (포트 9093)
ㄴ Broker 3 (포트 9094)
```

### 7. Offset (오프셋)

## 아키텍처

### 메시지 흐름

```
1. Producer가 메시지 발행
   ReservationService -> "예약생성 이벤트" -> Kafka

2. Kafka가 메시지 저장
   Partition에 메시지 저장 + 여러 브로커에 복제

3. Consumer가 메시지 구독
   Consumer1이 Partition0에서 이벤트 읽음

4. Consumer 처리 및 Offset 커밋
   메시지 처리 완료 후 offset 저장 (재시작 시 이어서 읽기)
```

## 장단점

### 장점

#### 1. 고성능 & 고처리량

```
- 초당 수백만 개 메시지 처리
- 배치 처리로 네트워크 효율성 극대화
- 비동기 처리로 응답 시간 개선
```

#### 2. 확장성 (Scalability)

```
- 파티션 추가로 처리량 증대
- 브로커 추가로 저장 용량 증대
- 컨슈머 그룹 확장으로 병렬 처리
```

#### 3. 내결함성 (Fault Tolerance)

```
- 복제 (Replication)로 데이터 손실 방지
- 한 브로커 장애가 서비스에 영향 없음
- 메시지 지속성 보장
```

#### 4. 메시지 순서 보장

```
- 같은 Key의 메시지는 같은 파티션으로
- 파티션 내 메시지 순서 보장
```

#### 5. 장기 보관 가능

```
- 메시지 영구 저장 가능 (디스크 용량 한도)
- 과거 데이터 재처리 가능
- 이벤트 로깅용 강력한 도구
```

#### 6. Decoupling (느슨한 결합)

```
- Producer는 Consumer와 무관
- Consumer 추가/제거가 시스템에 영향 없음
- 독립적인 확장 가능
```

### 단점

#### 1. 복잡성 증가

```
- 추가 컴포넌트 (Kafka 클러스터, ZooKeeper)
- 운영 리소스 증가
- 네트워크 복잡성
```

#### 2. 운영 오버헤드

```
- 클러스터 관리 필요
- 모니터링 및 알림 설정
- 성능 튜닝 필요
```

#### 3. 높은 초기 투자 비용

```
- 인프라 준비 (3개 이상 브로커)
- 학습곡선 가파름
- 기술 인력 필요
```

#### 4. 정확히 한 번 처리 (Exactly-once) 어려움

```
- 메시지 중복 가능성
- 컨슈머 장애 시 처리 가능성
- 멱등성 설계 필요
```

#### 5. 메모리/디스크 사용

```
- 메시지 저장으로 디스크 용량 많이 필요
- 클러스터 관리로 메모리 오버헤드
```

## 언제 Kafka를 도입할까?

### ✅ Kafka 도입하기 좋은 시점

```
1. 초당 처리 메시지가 10,000개 이상
2. 여러 시스템이 같은 이벤트를 개별 처리
3. 이벤트 재처리 가능성 필요
4. 장기 이벤트 로깅 필요
5. 마이크로서비스 간 느슨한 결합 필요
6. 실시간 데이터 파이프라인 필요
```

### ❌ Kafka 불필요한 경우

```
1. 초당 처리 메시지가 1,000개 미만
2. 간단한 요청-응답 패턴
3. 메시지 보관 불필요
4. 팀이 아직 작음 (운영 복잡도 감당 어려움)
```
